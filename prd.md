# ТЗ / Product Requirements Document

## MVP: AI-first IDE-like интерфейс для работы с Git-репозиторием и текстовыми документами

## 1. Цель продукта

Создать минимально жизнеспособную систему, похожую по UX на VS Code / IDE, в которой пользователь:

* подключает Git-репозиторий;
* видит дерево файлов слева;
* открывает и редактирует файлы в центральном редакторе;
* взаимодействует с AI-помощником справа либо через компактную плавающую командную панель;
* запускает AI-действия прямо в контексте открытого файла;
* получает результат как выделенный фрагмент в редакторе;
* может оставить результат в текущем файле либо вынести его в новый файл/новую вкладку.

MVP не придумывает новый паттерн взаимодействия, а повторяет знакомую механику IDE:

* sidebar с деревом файлов;
* tabs для открытых файлов;
* markdown/text editor в центре;
* AI assistant как docked panel или floating command input;
* действия применяются к текущему файлу, выделенному фрагменту или текущему контексту вкладки.

---

## 2. Границы MVP

### Входит в MVP

* Подключение локального или удалённого Git-репозитория.
* Отображение дерева файлов.
* Открытие файлов во вкладках.
* Редактирование текстовых файлов и Markdown.
* AI-команды над текущим файлом или выделением.
* Отображение AI-результата как inline diff / выделенный блок.
* Перенос результата в новый файл.
* Выбор одной из нескольких текстовых LLM.
* История команд в рамках текущей сессии/файла.
* Базовая система промптов и шаблонов действий.
* Локальный запуск системы.
* Простейший front-end на React/HTML/CSS.

### Не входит в MVP

* Генерация изображений.
* Работа с бинарными файлами.
* Совместное редактирование несколькими пользователями.
* Полноценный Git client UI (merge conflict resolution UI, rebase UI и т.д.).
* Voice input.
* Mobile-first интерфейс.
* Полноценная enterprise-auth система.
* Автономные долгоживущие агенты.
* Автоматическая индексация всего кодового графа уровня enterprise.

---

## 3. Основные продуктовые принципы

1. **Local-first MVP**: система должна запускаться локально у пользователя.
2. **IDE-first UX**: пользователь должен чувствовать, что работает в редакторе, а не в чате.
3. **AI как действие над документом**: AI работает над текущим текстом, а не только отвечает сообщениями.
4. **Минимум модальности**: результат появляется там, где работает пользователь — в редакторе.
5. **Прозрачность изменений**: все AI-изменения должны быть видимы, обратимы и контролируемы.
6. **Text-only first**: на первом этапе только текстовые модели и текстовые артефакты.

## 4. Пользователи и роли

### Основной пользователь

**Автор / редактор / исследователь / разработчик**, который работает с текстом в репозитории и хочет:

* быстро читать и редактировать документы;
* запускать AI-команды над текущим контекстом;
* переписывать, суммаризировать, структурировать, дополнять текст;
* создавать новые файлы из AI-результатов;
* держать всё в логике IDE.

### Роль для MVP

* Один локальный пользователь.

## 5. Jobs To Be Done

1. Когда я работаю с репозиторием документов, я хочу видеть структуру файлов и быстро открывать их, чтобы не терять контекст.
2. Когда я редактирую Markdown/текст, я хочу вызывать AI как команду поверх текущего текста, чтобы ускорить создание и переработку контента.
3. Когда AI возвращает результат, я хочу видеть его прямо в редакторе, чтобы сразу принять, отклонить или вынести в новый файл.
4. Когда я работаю с несколькими документами, я хочу иметь вкладки, чтобы быстро переключаться между контекстами.
5. Когда мне нужен другой стиль ответа, я хочу переключать модель, не меняя интерфейс работы.

## 6. User Stories

### Epic 1. Подключение репозитория

* Как пользователь, я хочу подключить Git-репозиторий, чтобы работать с его файлами в интерфейсе.
* Как пользователь, я хочу видеть дерево файлов, чтобы быстро находить нужный документ.

### Epic 2. Работа с редактором

* Как пользователь, я хочу открывать файлы во вкладках, чтобы работать с несколькими документами одновременно.
* Как пользователь, я хочу редактировать текст прямо в редакторе, чтобы использовать систему как обычную IDE.
* Как пользователь, я хочу выделять фрагмент текста, чтобы применять AI-команду только к выбранной части.

### Epic 3. Работа с AI

* Как пользователь, я хочу общаться с AI-помощником в боковой панели, чтобы формулировать задачу естественным языком.
* Как пользователь, я хочу свернуть AI-панель в компактную плавающую строку ввода, чтобы не терять рабочее пространство редактора.
* Как пользователь, я хочу запускать AI-команды как команды над текущим документом, чтобы результат применялся непосредственно в редакторе.
* Как пользователь, я хочу выбирать модель, чтобы использовать подходящую нейросеть под задачу.

### Epic 4. Применение результата

* Как пользователь, я хочу видеть результат AI как выделенный фрагмент или diff, чтобы контролировать изменения.
* Как пользователь, я хочу оставить результат в текущем файле, чтобы быстро внедрить правку.
* Как пользователь, я хочу вынести результат в новый файл, чтобы создать новый документ без ручного копирования.

### Epic 5. Управление контекстом

* Как пользователь, я хочу, чтобы AI знал текущий файл, выделение и ближайший контекст, чтобы ответ был релевантным.
* Как пользователь, я хочу видеть историю AI-команд по файлу, чтобы повторять или переиспользовать действия.

## 7. User Flow

### Flow 1. Подключение репозитория

1. Пользователь открывает приложение.
2. Нажимает «Подключить Git».
3. Указывает локальный путь к репозиторию или URL.
4. Система валидирует репозиторий.
5. Слева отображается дерево файлов.
6. По умолчанию открывается стартовый экран или README.md.

### Flow 2. Открытие и редактирование файла

1. Пользователь выбирает файл в дереве.
2. Система открывает файл в новой вкладке.
3. Пользователь редактирует текст.
4. Изменения помечаются как unsaved.
5. Пользователь сохраняет файл.

### Flow 3. Запуск AI-команды из правой панели

1. Пользователь открывает файл.
2. Пользователь выделяет фрагмент или оставляет курсор без выделения.
3. Пользователь вводит команду в AI-панели.
4. Система собирает контекст: файл, выделение, соседний текст, метаданные.
5. Система отправляет запрос в выбранную модель.
6. AI-результат возвращается и вставляется как предложенное изменение / выделенный блок.
7. Пользователь выбирает: accept / reject / create new file.

### Flow 4. Работа через плавающую командную строку

1. Пользователь сворачивает правую AI-панель.
2. Поверх/внизу центрального редактора появляется компактная command bar.
3. Пользователь вводит команду.
4. Результат применяется в активном редакторе.
5. История команды сохраняется в контексте файла.

### Flow 5. Вынесение результата в новый файл

1. Пользователь получает AI-результат в текущем файле.
2. Пользователь выбирает «Создать новый файл из результата».
3. Система предлагает имя файла и путь.
4. После подтверждения создаётся новый файл.
5. Новый файл открывается в новой вкладке.

## 8. Use Cases в формате Given / When / Then

### UC-01. Подключение локального Git-репозитория

**Given** пользователь находится на стартовом экране и у него есть локальный Git-репозиторий
**When** пользователь выбирает команду «Подключить Git» и указывает путь к папке
**Then** система валидирует наличие `.git`, загружает дерево файлов и отображает его в левой панели

### UC-02. Подключение удалённого Git-репозитория

**Given** пользователь находится на стартовом экране и имеет URL Git-репозитория
**When** пользователь вставляет URL и нажимает «Подключить»
**Then** система клонирует репозиторий локально во временную рабочую директорию и отображает дерево файлов

### UC-03. Открытие файла

**Given** дерево файлов отображено в sidebar
**When** пользователь нажимает на текстовый файл
**Then** файл открывается в центральной панели в новой или существующей вкладке

### UC-04. Редактирование файла

**Given** файл открыт в редакторе
**When** пользователь изменяет текст
**Then** система показывает, что документ изменён, и позволяет сохранить изменения

### UC-05. Запуск AI над выделением

**Given** файл открыт и пользователь выделил фрагмент текста
**When** пользователь вводит команду «перепиши кратко»
**Then** система отправляет в AI выделенный фрагмент и релевантный контекст, а результат отображает как предложенную замену выделения

### UC-06. Запуск AI над всем документом

**Given** файл открыт и выделения нет
**When** пользователь вводит команду «сделай структуру документа лучше»
**Then** система интерпретирует команду как действие над текущим файлом целиком и возвращает предложенное изменение документа

### UC-07. Работа через свёрнутую AI-панель

**Given** правая AI-панель свёрнута
**When** пользователь вызывает floating command bar и вводит инструкцию
**Then** команда исполняется относительно активного файла/выделения, а результат отображается в редакторе

### UC-08. Принятие AI-результата

**Given** AI вернул изменение в виде diff или выделенного блока
**When** пользователь нажимает Accept
**Then** система применяет изменение к документу и сохраняет это в локальном состоянии редактора

### UC-09. Отклонение AI-результата

**Given** AI вернул изменение в виде diff или выделенного блока
**When** пользователь нажимает Reject
**Then** система убирает предложенное изменение и возвращает документ в предыдущее состояние

### UC-10. Создание нового файла из результата

**Given** AI вернул фрагмент результата в текущем документе
**When** пользователь выбирает «Создать новый файл»
**Then** система создаёт новый файл, переносит туда результат и открывает его в новой вкладке

### UC-11. Переключение модели

**Given** пользователь находится в рабочем интерфейсе
**When** пользователь выбирает другую текстовую модель из списка доступных моделей
**Then** следующие AI-запросы отправляются в выбранную модель с тем же UX-потоком

### UC-12. Работа с несколькими вкладками

**Given** у пользователя открыто несколько файлов
**When** пользователь переключается между вкладками
**Then** система сохраняет состояние каждой вкладки, включая unsaved changes, локальный контекст и историю AI-команд

### UC-13. Ошибка модели

**Given** пользователь отправил AI-команду
**When** внешний AI-провайдер вернул ошибку или timeout
**Then** система показывает понятное сообщение об ошибке и не повреждает состояние редактора

### UC-14. Неподдерживаемый файл

**Given** пользователь нажал на бинарный или неподдерживаемый файл
**When** система определяет неподдерживаемый формат
**Then** она не открывает его в текстовом редакторе и показывает сообщение «Формат не поддерживается в MVP»

## 9. Функциональные требования

### 9.1. Git / Repository Layer

FR-1. Система должна поддерживать подключение локального Git-репозитория.

FR-2. Система должна поддерживать подключение удалённого Git-репозитория через clone.

FR-3. Система должна отображать дерево файлов и папок в левой панели.

FR-4. Система должна уметь обновлять дерево после создания/переименования/удаления файла.

FR-5. Система должна ограничивать MVP поддержкой текстовых файлов.

### 9.2. Editor Layer

FR-6. Система должна открывать файлы в центральном редакторе.

FR-7. Система должна поддерживать вкладки.

FR-8. Система должна поддерживать редактирование Markdown и plain text.

FR-9. Система должна показывать состояние unsaved changes.

FR-10. Система должна поддерживать выделение фрагмента текста.

FR-11. Система должна поддерживать сохранение файла на диск.

FR-12. Система должна позволять создавать новый файл.

FR-13. Система должна позволять перемещать AI-результат в новый файл.

### 9.3. AI Interaction Layer

FR-14. Система должна отображать AI-помощника в правой панели.

FR-15. Система должна позволять сворачивать AI-панель в компактную floating command bar.

FR-16. Система должна принимать AI-команды в свободной текстовой форме.

FR-17. Система должна определять область применения команды: выделение, весь файл или активная вкладка.

FR-18. Система должна отправлять в AI минимум следующие данные: текст выделения, ближайший контекст, путь файла, язык/тип файла, инструкция пользователя.

FR-19. Система должна поддерживать список текстовых моделей для выбора.

FR-20. Система должна сохранять историю AI-команд в пределах сессии.

FR-21. Система должна позволять повторить последнюю команду.

### 9.4. Result Application Layer

FR-22. Система должна отображать AI-результат как предложенное изменение, а не безусловно заменять текст.

FR-23. Система должна поддерживать действия Accept / Reject.

FR-24. Система должна поддерживать действие «Создать новый файл из результата».

FR-25. Система должна визуально выделять область, изменённую AI.

FR-26. Система должна сохранять исходный текст до применения AI-изменения.

### 9.5. Session / Context Layer

FR-27. Система должна хранить локальное состояние активного репозитория.

FR-28. Система должна хранить список открытых вкладок.

FR-29. Система должна хранить локальный контекст файла: последнее выделение, последние команды, последнюю модель.

FR-30. Система должна позволять восстановить рабочую сессию после перезапуска приложения (опционально для MVP+, желательно для MVP).

## 10. Нефункциональные требования

NFR-1. Интерфейс должен быть простым, знакомым и минималистичным, без перегруженной навигации.

NFR-2. Основные действия (открыть файл, переключить вкладку, сохранить файл, вызвать AI-команду) должны выполняться не более чем за 1–2 действия пользователя.

NFR-3. Время открытия текстового файла размером до 1 MB должно быть < 300 мс на локальной машине среднего класса.

NFR-4. Время отображения дерева репозитория до 5 000 файлов должно быть приемлемым для локальной работы (< 2 с при первичной загрузке).

NFR-5. AI-запрос не должен блокировать весь UI; редактор должен оставаться интерактивным.

NFR-6. Все AI-изменения должны быть обратимы.

NFR-7. Система должна быть локально разворачиваемой через простой сценарий запуска.

NFR-8. Архитектура должна позволять легко добавить новые текстовые модели и новые AI-команды.

NFR-9. Приложение должно быть безопасным по умолчанию: ключи моделей не должны утекать в UI-логи.

NFR-10. Должна поддерживаться базовая наблюдаемость: логирование ошибок, AI-запросов, времени ответа.

## 11. Информационная архитектура интерфейса

### 11.1. Layout

Экран делится на 3 основные зоны:

* **Левая панель** — дерево репозитория.
* **Центральная панель** — вкладки + редактор.
* **Правая панель** — AI assistant.

### 11.2. Режимы AI-панели

1. **Docked mode** — полноценная правая колонка с историей взаимодействия.
2. **Collapsed mode** — тонкая плавающая командная панель поверх центральной зоны.

### 11.3. Основные UI-компоненты

* Repository Tree
* Tabs Bar
* Editor View
* AI Assistant Panel
* Floating Command Bar
* Model Selector
* Diff / Suggestion Overlay
* Action Buttons: Accept / Reject / Create New File
* Status Bar

---

## 12. Техническая архитектура MVP

### 12.1. Общая схема

Система состоит из 5 логических слоёв:

1. **Frontend Layer** — React UI.
2. **Application API Layer** — backend/API, обслуживающий UI.
3. **AI Orchestration Layer** — подготовка контекста, шаблоны промптов, вызовы LLM.
4. **Repository & File Layer** — работа с Git, файловой системой и состоянием документов.
5. **Persistence Layer** — БД и локальное хранилище.

## 17. AI Orchestration Layer

Это ключевой слой, который отражает логику Given / When / Then в виде прикладных AI-действий.

### Ответственности слоя

* Определить, над чем выполняется команда:

  * выделение;
  * весь файл;
  * активный блок/раздел.
* Подготовить контекст.
* Выбрать шаблон промпта.
* Подставить системный и пользовательский prompt.
* Вызвать нужную модель.
* Преобразовать результат в формат suggestion/diff.
* Вернуть структуру, пригодную для применения в редакторе.

### Структура AI-запроса

```json
{
  "model": "gpt-4.1-mini",
  "action": "rewrite",
  "scope": "selection",
  "instruction": "Перепиши кратко и структурно",
  "file": {
    "path": "docs/spec.md",
    "type": "markdown"
  },
  "selection": {
    "start": 120,
    "end": 280,
    "text": "..."
  },
  "surrounding_context": {
    "before": "...",
    "after": "..."
  }
}
```

### Базовые типы AI-команд для MVP

* Rewrite
* Summarize
* Expand
* Structure
* Convert to bullets
* Convert to PRD/TZ style
* Clean up wording
* Continue writing
* Create new draft from current context

### Шаблоны системных промптов

#### Prompt 1. Rewrite selected text

**System prompt:**
«Ты редактор текста в IDE. Ты не ведёшь диалог ради диалога. Ты возвращаешь только результат редактирования текста для вставки в документ. Сохраняй смысл, соблюдай язык исходного текста, не добавляй пояснений вне результата.»

#### Prompt 2. Structure document

**System prompt:**
«Ты AI-ассистент внутри редактора документов. Твоя задача — преобразовывать текст в более структурный документ, пригодный для прямого использования. Верни только итоговый текст документа.»

#### Prompt 3. Create new file content

**System prompt:**
«Ты создаёшь новый документ на основании текущего контекста файла. Верни только содержимое нового файла без комментариев о процессе.»

### Prompt assembly logic

Формула:

* system prompt по типу действия;
* user instruction;
* file metadata;
* selected text или full file;
* optional surrounding context;
* response format instruction.

### Требование к формату ответа модели

Для MVP желательно требовать один из двух форматов:

1. **Plain text result** — если команда возвращает полную замену участка.
2. **Structured JSON** — если нужен контролируемый diff.

Пример JSON-ответа модели:

```json
{
  "mode": "replace_selection",
  "result_text": "...",
  "explanation": "optional for logs only"
}
```

Для UI пользователю explanation можно не показывать.

## 18. Логика сервисов по Given / When / Then

### Service: `ContextResolverService`

**Given** активный файл, текущее выделение, положение курсора и метаданные файла
**When** пользователь отправляет AI-команду
**Then** сервис определяет scope (`selection`, `full_file`, `section`) и собирает контекст для запроса

### Service: `PromptTemplateService`

**Given** тип действия и инструкция пользователя
**When** сформирован scope и контекст
**Then** сервис выбирает системный prompt и собирает финальный prompt-payload

### Service: `ModelRouterService`

**Given** пользователь выбрал модель
**When** собран AI-payload
**Then** сервис направляет запрос в соответствующий LLM provider adapter

### Service: `AiExecutionService`

**Given** есть prompt-payload и выбранная модель
**When** пользователь запускает команду
**Then** сервис исполняет запрос, обрабатывает ошибки и возвращает нормализованный результат

### Service: `SuggestionService`

**Given** AI вернул текстовый результат
**When** результат получен backend
**Then** сервис формирует suggestion object, содержащий original, result, diff и привязку к файлу/выделению

### Service: `EditorApplyService`

**Given** есть suggestion object
**When** пользователь нажимает Accept
**Then** сервис применяет изменение к нужному файлу и обновляет состояние вкладки

### Service: `CreateFileFromResultService`

**Given** есть suggestion/result
**When** пользователь выбирает создание нового файла
**Then** сервис создаёт новый файл, записывает туда результат и открывает его в новой вкладке

### Service: `CommandHistoryService`

**Given** пользователь выполнил AI-команду
**When** выполнение завершилось success/error
**Then** сервис сохраняет запись в истории команд

## 19. Выбор моделей

MVP должен поддерживать абстракцию текстовой модели, а не жёсткую привязку к одному провайдеру.

### Интерфейс модели

```ts
interface TextModelProvider {
  name: string;
  execute(input: AiExecutionInput): Promise<AiExecutionOutput>;
}
```

### Возможные провайдеры для MVP

* OpenAI
* Anthropic
* Google
* OpenRouter (как мульти-провайдерный шлюз)
* Local model adapter (опционально)

### Обязательное поведение

* единый контракт запроса;
* единый нормализованный формат ответа;
* логирование latency и ошибок по каждой модели.

## 20. Инфраструктурный контур

Поскольку MVP будет локальным, инфраструктура должна быть максимально простой.

## 21. Telegram bot слой (отдельно, как инфраструктурный/операционный интерфейс)

Если нужен локальный Telegram bot как дополнительный интерфейс управления, его лучше рассматривать **не как основной UI**, а как вспомогательный канал.

### Что может делать Telegram bot в MVP

* Запуск/остановка локального приложения.
* Получение статуса backend.
* Просмотр списка доступных репозиториев.
* Отправка короткой AI-команды над последним активным файлом.
* Получение результата в текстовом виде.

### Что Telegram bot не должен заменять

* Основной редактор.
* Основной UX по diff/accept/reject.
* Навигацию по файловому дереву уровня IDE.

### Вывод

Основное приложение — web frontend. Telegram bot — опциональный control plane / remote trigger interface.

## 22. Безопасность

* API-ключи моделей хранятся только на backend.
* Frontend не должен получать raw secrets.
* Нужна базовая защита от path traversal при работе с файлами.
* Репозиторий должен открываться только внутри разрешённой workspace-директории.
* Все операции записи должны быть явными и журналируемыми.

---

## 23. Ограничения и компромиссы MVP

* Нет сложного семантического поиска по всему репозиторию.
* Нет долговременной памяти между множеством проектов, кроме истории команд и сессий.
* Нет совместной работы.
* Нет полноценного визуального Git UI.
* Нет advanced agent workflows.
* Основной фокус: **редактирование текста с AI внутри IDE-подобного UX**.

---

## 24. Критерии приёмки MVP

MVP считается готовым, если:

1. Можно подключить Git-репозиторий.
2. Видно дерево файлов слева.
3. Файлы открываются в табах.
4. Markdown/текст можно редактировать.
5. AI-панель работает в правом режиме.
6. AI-панель можно свернуть в компактную command bar.
7. AI-команда применяется к выделению или ко всему файлу.
8. Результат отображается как предложенное изменение.
9. Можно принять/отклонить изменение.
10. Можно вынести результат в новый файл.
11. Можно выбрать модель из списка текстовых моделей.
12. Всё запускается локально и не требует сложной инфраструктуры.

## 26. Короткое техническое резюме

Для MVP достаточно следующего стека:

* **Frontend:** React + TypeScript + Monaco Editor
* **Backend:** Node.js + Fastify/Express
* **Git/File:** simple-git + fs
* **DB:** SQLite
* **AI:** provider adapters для текстовых LLM
* **Deploy:** local run / docker-compose

Ключевая архитектурная идея:
**AI не существует отдельно как чат-приложение; AI — это слой действий над активным документом внутри редактора.**
